# Object Modeling for Design

把 Narrative Object Model（Heidi Adkisson 提出）当作一把解剖刀，用在设计评审和设计系统决策上。

## 为什么需要这个技能

组件级设计系统不足以保证体验一致性。B 端与平台型产品的典型症状——功能臃肿、交互不一致、动作散落、词汇混乱、"feature tack-on"——根源都在**对象层没有被显式建模**，而不是组件样式不统一。

对象模型提供的是**结构视图**，和 user story/journey map 这类**任务视图**互补。两者都重要，但设计稿评审阶段，结构视图被严重低估。

## 核心概念速览

- **Object（对象）**：系统里的名词。Account、Tweet、Order、Workflow、Tenant、Role 都是。区分"对象"（概念）和"实例"（某个具体的 Account）。
- **Action（动作）**：系统里的动词，附着在对象上。Create / Read / Update / Delete 是最基本的四类，外加对象特有的动作（Publish、Assign、Approve）。
- **Attribute（属性）**：对象的数据字段。Title、Status、Owner、CreatedAt。
- **Relationship（关系）**：四种
  - **Association（关联）**：普通连线，A 和 B 有关。例 Account 发 Direct Message。
  - **Aggregation（聚合）**：A 是 B 的集合。空心菱形。例 Moment 聚合 Tweets。
  - **Component（组件/依赖）**：B 依存于 A，A 没了 B 也不存在。实心菱形。例 Hashtag 依存于 Tweet。
  - **Inheritance（继承）**：父子，子对象继承父的特征。空心箭头。例 AdminAccount / UserAccount 继承 Account。

这四种关系的判别细节见 `references/relationships.md`，一旦在分析中遇到"这俩算什么关系"的犹豫就查它。

## 工作流：拿到设计稿后怎么办

按这个顺序推进，不要跳步，尤其不要过早跳到"给改进建议"。前面做不扎实，后面的建议就是无根之木。

### 步骤 1：识别对象（Objects）

扫过设计稿上的所有**菜单项、按钮、列表标题、表单标签、面包屑、标签页、空状态文案**。这些是对象和动作的浓缩信号源。

- 菜单里反复出现的名词，很可能是对象
- 列表视图的每一行代表什么，那就是对象
- 表单在创建什么，那个东西就是对象
- 导航侧栏的主要入口，基本对应顶层对象或对象集合

对 B 端/平台型产品尤其注意以下常见对象原型，完整清单见 `references/enterprise-patterns.md`：

- 主体类：Tenant/Organization、Workspace、Project、Team、User、Role、Permission
- 业务实体类：Order、Ticket、Task、Document、Asset
- 流程类：Workflow、Approval、Stage、Transition
- 资产类：Template、Policy、Rule、Configuration
- 日志类：AuditLog、Activity、Notification

输出：一份带简短说明的对象清单。每个对象一两句解释它是什么、它的实例是什么。这一步必须先做完再走下一步。

### 步骤 2：标记核心对象（Core / Signature Objects）

不是所有对象同等重要。核心对象是承载产品价值主张的那一两个，应当优先设计、并成为设计系统的 signature elements。

判断标准：

- 如果从系统里抽掉它，产品就不再是同一个产品
- 用户的主要时间花在和它交互
- 产品的核心指标围绕它定义

为什么重要：B 端产品常见错误是把 signature object 的设计等同于普通 CRUD，结果用默认后台样式堆了个没有产品性的系统。核心对象值得专门的卡片样式、详情页布局和空状态表现。

输出：对象清单上把核心对象标粗。

### 步骤 3：梳理对象关系

对每一对有关联的对象，先写一句自然语言描述，再判断属于四种关系里的哪种。

写法参考：

- "一个 **Project** 可以包含多个 **Task**"（aggregation）
- "一个 **Comment** 依存于一个 **Task**"（component）
- "**AdminUser** 是一种特殊的 **User**"（inheritance）
- "一个 **User** 可以订阅一个 **Project**"（association）

对平台型产品，**多租户关系**几乎总是最容易忽略又最关键的：每个业务对象是否都明确归属于某个 Tenant/Workspace？跨租户的对象如何处理？这一层不梳理清楚，后面的权限设计必然混乱。

输出：一段结构化文字，或当对象数大于等于 5 且关系复杂时，一张 Mermaid 图。图的用法见“可视化”部分。

### 步骤 4：填充 Actions 与 Attributes

对每个对象列出它的动作和属性。动作用 `actionName()` 的形式，便于视觉区分。

关键检查：

- **CRUD 完整性检查**：Create / Read / Update / Delete 是否被遗漏？遗漏是有意还是漏掉的？
- **相似对象的动作对齐**：功能相似的对象，动作清单应该结构化对齐
- **动作散落检查**：同一个对象的动作散落在多个地方，是 B 端产品里非常常见的可用性问题
- **数据口径交叉验证**：Dashboard / Overview 类界面的数字卡、summary 条和汇总指标，必须横向核对，尤其留意重复数字、状态撞号、对齐不一致和不同源数据的问题

把这些都列为结构性发现里的问题，因为它们既是体验 bug，也是对象建模 bug 的外显。

### 步骤 5：导出设计系统 / 组件 / 界面决策

前面的模型用来支撑以下四类决策：

**A. 设计系统抽象决策**

- 哪些对象该共用一个 Card 组件
- 哪些对象该做成独立组件
- 哪些对象可以通过继承复用

**B. 信息架构 / 导航决策**

- 导航的顶级入口应当对齐核心对象和顶级 aggregation，而不是功能动词
- Component 关系的对象不应当出现在顶级导航

**C. 一致性 / 词汇决策**

- 同一对象在不同页面是否用同一个名字
- 同一动作的按钮文案是否统一

**D. 功能完整性 / 界面改进建议**

- 基于 CRUD 检查指出缺失动作
- 基于相似对象对齐指出功能不对称
- 基于动作散落指出该归位的按钮

## 输出格式

默认以文字分析为主，按下面的结构组织。不要一上来就给“界面改进建议”。

```text
## 对象识别
[列出识别到的对象，每个一两句说明，核心对象加粗]

## 对象关系
[结构化文字描述；对象多或关系复杂时附 Mermaid 图]

## 动作与属性（按对象分）
[列出每个对象的动作和关键属性，标出 CRUD 缺口]

## 结构性发现
[按严重程度排列：
- 🔴 阻塞性问题
- 🟡 一致性问题
- 🟢 优化建议]

## 设计系统 / 组件建议
[基于对象模型导出的抽象建议]

## 具体界面改进
[给出足够具体的调整方向]
```

## 何时画图

不要每次都画图。对象少于 5 个、关系简单时，文字通常更清晰。以下情况适合 Mermaid `classDiagram`：

- 对象大于等于 5 个
- 关系超过两层嵌套
- 用户明确要求“画出来”
- 有继承层级

模板见 `assets/mermaid-template.md`。

## 容易踩的坑

- 把页面当对象
- 把组件当对象
- 过度建模
- 跳过核心对象判断
- 忽略多租户 / 权限维度

## 默认追问清单

每次分析时，主动对照以下问题：

1. 这是双边/多边平台吗？当前界面属于哪一端？
2. 这是多租户产品吗？
3. 同一端里有多少种 Role？
4. 核心对象用户自己怎么称呼？
5. 关键 component 对象是否存在并被独立建模？
6. 状态机是硬编码还是可配置？

如果设计稿信息明显不足：

- 先基于已有内容做部分模型，显式标出“已知”和“推断”
- 不要凭想象补全
